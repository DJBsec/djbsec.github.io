Which architecture pattern fits best?

This affects SEO, maintainability, and complexity. Pick your preferred approach.

A

Single Page + Modals

All conference data lives in _data/cybersecurity_calendar.yml. One page at /cybersecurity-calendar/ loads everything. Clicking a conference opens a modal with details.

Pros

  • Simple to build & maintain
  • Fast (one page)
  • Easy data refresh

Cons

  • No individual conference URLs
  • Weaker SEO for individual events
  • All JS must load upfront
B

Collection + Detail Pages

Each conference is a file in _conferences/. Jekyll generates /cybersecurity-calendar/defcon-2026/ etc. Calendar page links out to detail pages.

Pros

  • Clean SEO-friendly URLs per event
  • Schema.org markup per page
  • Shareable deep links

Cons

  • 50+ individual files to manage
  • Script must generate collection files
  • Slower build time
C

Hybrid (Recommended)

Single _data/cybersecurity_calendar.yml as source of truth. Calendar page with quick-view modals and a "Full Details" link per event that goes to a generated detail page at /cybersecurity-calendar/[slug]/.

Pros

  • Best SEO (individual URLs)
  • Single data source (easy to update)
  • Fast preview via modal
  • Deep linking & sharing

Cons

  • Most complex to build
  • Requires a layout for detail pages
